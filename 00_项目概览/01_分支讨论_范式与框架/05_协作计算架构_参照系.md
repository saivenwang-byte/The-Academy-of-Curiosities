# 协作计算架构 · 参照系（Agent 协作的「计算机架构」）

> **Status**: v1 自检加强 · Agent 自主维护  
> **用途**: 用 **真实计算机架构** 的语言，把 L1–L4、A1/A2/A4 从「比喻」变成 **可操作的参照系**  
> **范围**: 纯范式 — 不绑定任何具体 IP、卷次、仓库路径  
> **维护**: 发现透镜说不清时，先改本文，再改索引

---

## 0. 为什么要用「计算机架构」

多 Agent 长期协作，本质上是一个 **分布式系统**：

- 有 **指令**（Skill / Rule）  
- 有 **存储层次**（会话 / 缓冲 / 正典）  
- 有 **特权级**（讨论区 vs 执行区）  
- 有 **安全边界**（什么能写、什么只能读）  
- 有 **优化循环**（从运行 trace 升格为新 Skill）

用架构语言的好处：**每个透镜都有标准问法**，不会停在「感觉上应该分层」。

---

## 1. 总线模型：五层 + 两条平面

```
                    ┌─────────────────────────────────┐
  控制平面          │  范式讨论 · 定案 · 升格        │  ← A2 讨论层
  (Control Plane)   └──────────────┬──────────────────┘
                                   │ 定案下发（只读契约）
                    ┌──────────────▼──────────────────┐
  数据平面          │  L1 计算 │ L2 存储 │ L3 隔离   │  ← 执行层
  (Data Plane)      │         L4 优化反馈              │
                    └─────────────────────────────────┘
```

| 平面 | 架构类比 | 协作含义 | 主透镜 |
|------|----------|----------|--------|
| **控制平面** | 路由协议、策略表、配置管理 | 定框架、定边界、定关闭条件 | **A2** E3 |
| **数据平面** | CPU + 内存 + I/O | 实际写稿、改图、跑脚本 | A2（执行侧） |
| **反馈环** | PMU / profiling → PGO | 会话教训 → Skill 升格 | L4 B4 |

**A2 加强定义（控制 vs 数据）**：

| 维度 | 讨论层（控制平面） | 执行层（数据平面） |
|------|-------------------|-------------------|
| **产出** | 命名、边界、路由表、关闭条件 | 文件 diff、成图、lint 结果 |
| **变更频率** | 低（框架 v1 冻结后少改） | 高（日常生产） |
| **失败代价** | 概念漂移、无限开会 | 错误写入、返工 |
| **谁签字** | 主编 / 架构决策者 | 流水线 + 人审节点 |
| **自检** | 离开本区不必改任何执行文件 | 必须改 `[项目]` 文件才能闭环 |

**硬规则 CP-1**：控制平面 **不直接写** 数据平面的 mutable 状态；只下发 **契约**（Rule / Skill / SOP / lint）。

## 2. L1 · 计算层（Skill 体系 ≈ ISA + 专用协处理器）

### 2.1 架构映射

| 计算机概念 | 协作栈对应 | 说明 |
|------------|------------|------|
| **ISA（指令集）** | Rule + 路由表 | 永远合法的「指令格式」 |
| **通用 CPU** | 主 Agent / 主编对话 | 调度、分支预测、异常处理 |
| **协处理器** | 专精 Skill（审/写/测/…） | 单一场景峰值性能 |
| **系统调用** | `@skill` 触发 / MCP / 脚本 | 用户态 → 受控能力 |
| **中断** | Hook（stop/start） | 异步插入检查 |

### 2.2 L1 三条不变量（加强）

| ID | 不变量 | 违反症状 |
|----|--------|----------|
| **L1-I1** | **单入口路由** | 同一任务多个 Skill 各写一段 → 冲突 |
| **L1-I2** | **专精不越界** | 视觉 Skill 改正典人名 |
| **L1-I3** | **无 Skill 则停** | 未匹配路由时禁止 silent 硬写 |

### 2.3 路由判定表（操作化 · 中性）

| 请求类型 | 路由目标（抽象） | 禁止 |
|----------|------------------|------|
| 拆框架 / 定边界 | 控制平面 · 范式讨论 | 直接改 `[产物]` |
| 写 `[正文]` | `[创作 Skill]` | 无任务包硬写 |
| 审 `[成图]` | `[视觉 Skill]` | 通用 Agent 独断 PASS |
| 改 `[脚本]` | `[工程 Skill]` + lint | 无 review 合并 |
| 学外部系统 | `[学习卡片]` 模板 | 跳过「不学什么」 |

### 2.4 L1 开放题（已部分自答）

**Q: 60+ 专精 Agent 按什么分类？**  
**自答 v1**：**双轴** — 先 **职能**（路由/审/写/测/学），再 **领域**（可选挂载）。  
域是插件，职能是总线；避免 60×N 全连接。

## 3. L2 · 存储层（记忆 ≈ 存储层次结构）

### 3.1 架构映射（经典 memory hierarchy）

```
  寄存器级     当前 turn 上下文          ~KB   易失   最快
      ↓
  L1 Cache     会话工作缓冲              ~10KB  半持久  快
      ↓
  RAM          决策记录 / 任务包草稿      ~100KB 可版本化
      ↓
  磁盘         Git 正典 / docs / skills  MB+   持久   慢
      ↓
  冷存储       归档 / transcript         GB    只读   最慢
```

| 层级 | 协作对象 | A4 透镜问法 | 升格判据 |
|------|----------|-------------|----------|
| 寄存器 | 本轮 messages | 不持久 | — |
| L1 Cache | `working_buffer` 类 | 重启可丢吗？ | 丢则降效，不丢正典 |
| RAM | 决策记录 DRAFT | 多场讨论共享？ | 定案 → 磁盘 |
| 磁盘 | Rule / Skill / 正典 | 谁可 Write？ | L3 特权 |
| 冷存储 | 归档讨论 | 只读引用 | 永不回写 |

### 3.2 L2 三条不变量（加强）

| ID | 不变量 | 说明 |
|----|--------|------|
| **L2-I1** | **单写者** | 同一正典文件同一时刻只有一个「主写 Skill」 |
| **L2-I2** | **显式升格** | Cache → 磁盘必须经控制平面定案或 lint PASS |
| **L2-I3** | **可重启** | 新会话读磁盘 + RAM 摘要即可恢复，不依赖寄存器 |

### 3.3 A4 加强：永久 vs 工作记忆判据

| 写入目标 | 条件 | 反例 |
|----------|------|------|
| **磁盘（正典）** | 跨会话必用 + 多 Agent 共识 + 关闭条件满足 | 单场 brainstorm |
| **RAM（决策记录）** | 多场连续讨论 + 未冻结 | 已定案重复写 |
| **Cache（缓冲）** | 仅接续下一会话 | 当正典用 |

## 4. L3 · 隔离与安全（≈ MMU + 特权环 + 沙箱）

### 4.1 架构映射

| 计算机概念 | 协作栈对应 |
|------------|------------|
| **Ring 0 / 内核** | 人类主编 + 不可 AI 单改的正典 |
| **Ring 3 / 用户态** | Agent 日常 Write 区 |
| **MMU** | Rule + Hook：地址 = 路径，权限 = R/W/X |
| **AgentShield** | 执行前 syscall 拦截 |
| **Capability** | Skill 白名单（能触哪些工具/路径） |

### 4.2 特权环模型（L3 加强 · 范式通用）

```
  Ring 0   主编签字 · L0 契约（框架冻结、伦理）     AI: 禁 Write
  Ring 1   控制平面产出（SOP、路由表、学习卡片）    AI: 提案 → 人审
  Ring 2   执行正典（Skill、Rule、lint）           AI: Write + lint
  Ring 3   草稿区（缓冲、DRAFT、实验分支）         AI: 自由 Write
```

| 操作 | 最低环 | 无 Shield 时替代 |
|------|--------|------------------|
| 改 Ring 0 | 主编 | 文档 + 禁止 Rule |
| 升格 Ring 1→2 | 控制平面关闭条件 | 决策记录签字 |
| Ring 3→2 合并 | lint + 人审 | pre_push_check |
| 破坏性 syscall | Hook 确认 | User Rule |

### 4.3 L3 三条不变量（加强）

| ID | 不变量 |
|----|--------|
| **L3-I1** | **默认拒绝**：未在白名单的路径 / 工具 = 不可 Write |
| **L3-I2** | **讨论区 = Ring 1**：范式区文档不得含执行 checklist（E3） |
| **L3-I3** | **可审计**：每条拦截能说出规则 ID（非黑盒） |

### 4.4 A2 × L3 联合：防火墙规则 FWP

| 规则 ID | 若检测到… | 则… |
|---------|-----------|-----|
| FWP-1 | 范式区出现 `[卷]`/`[角色]` 真名 | 退回执行区 |
| FWP-2 | 执行 Agent 改 Ring 0 | 停止 + 提案模式 |
| FWP-3 | 无路由硬写 Ring 2 | 要求补 Skill 触发 |
| FWP-4 | L4 自动写 Ring 2 | 禁止；只写 Ring 3 队列 |

## 5. L4 · 优化层（≈ Profile-Guided Optimization）

### 5.1 架构映射

| 计算机概念 | 协作栈对应 |
|------------|------------|
| **Profiling** | 会话 retro、失败 lint、返工 trace |
| **Hot path** | 高频触发的 Skill / 规则 |
| **PGO / JIT** | 教训升格为新 Skill 或 Rule 补丁 |
| **Firmware update** | 框架 v1 → v1.1（控制平面） |
| **GC / prune** | 过时 Skill、半衰期复核（B4） |

### 5.2 L4 升格流水线（加强 · 四站）

```
 trace（执行/讨论中的失败或返工）
   → 候选（Ring 3 队列，自动或人工提交）
   → 评审（控制平面，D1 三段式 + 人审）
   → 升格（Rule / Skill / SOP / lint 之一）
   → 度量（下 N 次会话是否减少同类失败）
```

| 产出类型 | 写入环 | 须人审 |
|----------|--------|--------|
| 禁止清单 | Ring 2 Rule | 是 |
| 新 Skill | Ring 2 | 是 |
| SOP 修订 | Ring 1→2 | 是 |
| lint 规则 | Ring 2 脚本 | 是 |
| 学习卡片 | Ring 1 | 否（讨论区） |

### 5.3 L4 三条不变量（加强）

| ID | 不变量 |
|----|--------|
| **L4-I1** | **无 trace 不升格** | 没有可引用失败例，不进评审 |
| **L4-I2** | **默认人审** | 自动提取只进 Ring 3 队列 |
| **L4-I3** | **可回滚** | 升格后 N 会话内可 revert 到 v-1 |

## 6. 透镜 × 架构 速查（A1 A2 A3 A4 加强版）

| 透镜 | 架构问法（标准句） | 答不了则 |
|------|-------------------|----------|
| **A1** 机器 vs 产品 | 「数据平面未 PASS，控制平面是否允许宣称 `[产品]` 成立？」 | 先 L1–L3 |
| **A2** 讨论 vs 执行 | 「这条变更改的是 **路由表** 还是 **运行时状态**？」 | 路由表→控制平面 |
| **A3** 学习 vs 复制 | 「这是 **firmware 原则** 还是 **配置文件粘贴**？」 | 写三段式 |
| **A4** 永久 vs 工作 | 「若明天 session 清零，这条知识从哪一层恢复？」 | 定存储层 |

## 7. 自检清单（Agent 每会话可用 · 不含项目名）

开始执行向工作前，Agent **自问**（任一条 NO → 退回控制平面）：

- [ ] **A2** 本场是控制平面还是数据平面？  
- [ ] **L1** 任务是否命中路由表？  
- [ ] **L2** 写入目标属于 Cache / RAM / 磁盘哪层？  
- [ ] **L3** 目标路径在哪个 Ring？有无 Write 权？  
- [ ] **L4** 若发现重复失败，是否应提交 trace 而非硬改？

## 8. 与 ECC 四层的精确同构（非 1:1 部署）

| ECC 层 | 本参照系 | 计算机等价 |
|--------|----------|------------|
| L1 技能 | §2 计算层 | ISA + 协处理器 + syscall |
| L2 记忆 | §3 存储层 | Cache → RAM → Disk |
| L3 安全 | §4 隔离层 | Ring + MMU + Shield |
| L4 学习 | §5 优化层 | Profiling → PGO |

**同构定理（范式 v1）**：  
ECC 四层 = **数据平面子系统**；A2 控制平面 = **其上的策略层**。  
缺控制平面 → 四层再全也会 **配置漂移**。
