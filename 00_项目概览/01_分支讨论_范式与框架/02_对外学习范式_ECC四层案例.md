## 1. 我们为什么拿 ECC 当第一个案例

ECC 代表一种 **「长期项目 + 多 Agent」** 的产品化思路：  
不是买一个更强的模型，而是买 **分工、记忆、安全、进化** 四件事。

适合本区，因为可以 **只谈范式**，不必绑定任何具体 IP 的卷次或角色。

---

## 2. 学习三段式（总表）

### [ECC] · 四层协作栈

**学什么**

- **专精分工**：大量场景化代理 > 单 Agent 包打天下（路由 + 执行分离）  
- **会话外记忆**：Hooks 把「工作态」持久化，降低重启成本  
- **执行前安全**：默认拦截高危操作，而非事后 git revert  
- **经验产品化**：会话产物可升格为 Skill/模式，形成复利  

**不学什么**

- **60+ 全量复制**：不意味着每个外部代理都要装进每个仓库  
- **黑盒 AgentShield**：若无法审计规则，不能替代明确的边界文档  
- **全自动升格**：未经人审的模式提取，可能污染正典  
- **替代人类主编审美**：创作终审、品牌、伦理不能交给 L4 自动提取  

**为什么**

- 长期 IP 项目的瓶颈常在 **协作连续性**，不在单次生成质量  
- 我们已有「文件正典 + Skill 子集」，缺的是 **统一的四层语言和跨会话机制**  

**抽象层**

- 命名：**协作栈四层** — L1 分工 · L2 记忆 · L3 安全 · L4 学习

## 3. 四层 · 仅范式定义（无项目绑定）

```
L4  持续学习   讨论/执行中的可复用模式 → 升格为 Skill/SOP/Rule
L3  安全扫描   执行前拦截；讨论边界防火墙
L2  记忆持久   跨会话接续；正典 vs 工作缓冲
L1  技能体系   场景化代理 + 路由表
```

| 层 | 范式问题（在本区讨论） | 不在本区讨论 |
|----|------------------------|--------------|
| L1 | 专精 Agent 如何分类？路由表长什么样？ | 某个 Skill 的 prompt 字句 |
| L2 | 什么必须文件化？什么可 session 缓冲？ | 某次会话写了哪段代码 |
| L3 | 高危操作清单如何设计？讨论区防火墙？ | 某文件 chmod 细节 |
| L4 | 升格 SOP：人审节点放哪？ | 某卷教训原文 |

## 4. 同构映射（讨论用 · 非部署结论）

外部 **生产栈** 与内部 **讨论栈** 可同构：

| ECC 生产栈 | 本讨论区讨论栈 | 映射关系 |
|------------|----------------|----------|
| L1 技能体系 | L1 分工路由 | 谁负责「想框架」 |
| L2 记忆持久 | L2 记忆接续 | 决策记录 + 缓冲 |
| L3 AgentShield | L3 边界安全 | 本区黑名单 |
| L4 持续学习 | L4 学习沉淀 | 三段式学习卡片 |

**讨论结论（范式层，待主编确认）**：

> 同构 **有助于记忆和沟通**，但不意味着两套栈必须 **同一实现**（例如讨论区不必装 AgentShield）。

## 7. 开放讨论题（纯范式）

1. **L1**：专精 Agent 的分类法 — 按 **职能**（审/写/测）还是按 **领域**（文化/视觉/代码）？  
2. **L2**：工作记忆默认 **版本化** 还是 **本地缓冲**？判据是什么？  
3. **L3**：安全是 **规则文档** 还是 **执行 Hook** 优先？能否两层并存？  
4. **L4**：「学习」的产出物类型枚举 — Skill / Rule / SOP / lint / 禁止清单？
